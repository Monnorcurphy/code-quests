## Findings

### CRITICAL — `specs/bugs/review-1.md`
Mock fixtures in `capture-walkthrough.spec.ts` violate `QuestSchema`, `FailureSummarySchema`, `InputRequestSchema`, and `AgentEvent('monster_appeared')`. The client parses every API response with Zod (`fetchJson` in `packages/client/src/lib/api.ts:135`), so the React Query hooks will error out and the demo UI never renders. The 12 captured PNGs will show empty / error states rather than the showcase narrative described in `specs/done/phase-11-walkthrough.md`, defeating the task's primary deliverable.

Specific violations:
- `currentScene` uses unprefixed enum values (`'dungeon'`, `'cave'`, `'forest'`, or `null`) when the schema requires `'quest-*'`.
- `specAudit` is missing required `runAt` field.
- `failureSummary.recommendation` uses `'repost_with_new_equipment'`, which is not in the valid enum (`'retry' | 'repost_with_clarification' | 'retire' | 'break_into_smaller' | 'level_up_first'`).
- `setInputRequest` payload uses `{prompt, requestedAt}` instead of the schema's `{question, awaitingSince}`.
- `monster_appeared` events omit the required `encounterId`.

## INFORMATIONAL notes

- The capture spec never calls `expect()`. This matches its purpose as a deterministic image-capture utility rather than a behavioral test, but it does mean the spec will pass silently with broken screenshots. The bug fix (review-1) includes a recommendation to add at least one assertion per step as a sanity gate. Treat assertion additions as a future polish item beyond the strict bug scope.
- The walkthrough doc's "Re-post with new equipment" CTA wording diverges from the canonical recommendation enum values; once review-1 is fixed to use `'repost_with_clarification'`, the visible label on the post-mortem panel may differ from the walkthrough copy. Either keep the marketing copy and add a UI alias in a follow-up task, or update the walkthrough copy to match what the panel actually renders.
- `assets/screenshots/phase-11/.gitkeep` is committed but `.gitkeep` is not declared as a directory-marker pattern in the repository conventions I could verify. It's an industry-standard convention so this is acceptable.
- `main.tsx` window injection wraps the new stores in `import.meta.env.DEV` — matches the existing `__townStore` pattern. ✅
- The capture spec uses `Parameters<typeof test.fn>[0]['page']` for typing — slightly fragile but compiles under TS strict mode. Not worth filing.
